# Methodology

## Overview

Each server is benchmarked in isolation against the same workload, on the same hardware, with the same loaded terminology data. Results are comparable across runs as long as the hardware and data versions remain the same.

## Test environment

- **FHIR version:** R4
- **Network:** all servers and the k6 load generator run on the same host — loopback only, no network latency
- **Isolation:** one server running at a time, no concurrent workloads

## Two-phase approach

### Phase 1 — Preflight

Before benchmarking, a correctness check verifies each server's response for each test case. A test *passes* preflight if the server returns a semantically correct response — right resource type, expected parameters, correct values.

Tests that fail or are not supported are excluded from that server's benchmark run and recorded in the **capability matrix**.

### Phase 2 — Benchmark

Only preflight-passing tests are benchmarked. Each test is run at three concurrency levels:

| Level | Virtual users | Duration |
|-------|--------------|----------|
| Low   | 1 VU  | 30 s |
| Mid   | 10 VUs | 30 s |
| High  | 50 VUs | 30 s |

A **warm-up pass** (5 s at 10 VUs, discarded) runs before measurement to allow JIT compilers and connection pools to reach steady state.

## Input pools

Each test draws inputs randomly from a pool of 2,000+ entries per code system. This ensures a realistic working-set scenario rather than testing the same code repeatedly. Pools are harvested from the terminology data loaded into the server before the run.

## Metrics

| Metric | Description |
|--------|-------------|
| `rps` | Throughput — requests per second |
| `p50 / p95 / p99` | Request latency percentiles |
| `error_rate` | Fraction of requests that returned an HTTP error or failed |
| `mem_used_bytes` | Server memory footprint at idle |
| `data_volume_bytes` | On-disk size of the terminology data |

## Composite score

Each server receives a single composite score in the range $[0, 100]$ where $100$ is the top performer in the run.

### Step 1 — Effective RPS

For each *(server, test)* pair, compute the effective RPS at each VU level and take the maximum across levels:

$$
\text{effRPS}_{s,t} = \max_v \bigl( \text{rps}_{s,t,v} \times (1 - \text{errorRate}_{s,t,v}) \bigr)
$$

Multiplying by $(1 - \text{errorRate})$ penalises servers that sustain throughput by returning errors.

### Step 2 — Normalising weight

Tests naturally differ in throughput — a simple lookup saturates faster than a complex expansion. Without adjustment, high-throughput tests would dominate the score. We use a **TechEmpower-style normalising weight** that scales each test's contribution to the same RPS unit as the reference test (LK01 — simple SNOMED lookup):

$$
w_t = \frac{\overline{\text{effRPS}}_{\text{LK01}}}{\overline{\text{effRPS}}_t}
$$

where $\overline{\text{effRPS}}_t$ is the mean effective RPS for test $t$ across all servers in the run.

### Step 3 — Weighted RPS

For each *(server, test)* pair, apply the normalising weight and a per-test **bias** coefficient $b_t \in (0, 1]$:

$$
\text{wRPS}_{s,t} = \text{effRPS}_{s,t} \times w_t \times b_t
$$

Bias coefficients down-weight test categories that are less representative of real-world workloads (e.g. `FS` FHIR search has $b = 0.4$). Default is $1.0$.

### Step 4 — Raw score

Sum weighted RPS across all tests:

$$
\text{rawScore}_s = \sum_t \text{wRPS}_{s,t}
$$

### Step 5 — Normalise to 0–100

$$
\text{score}_s = \frac{\text{rawScore}_s}{\max_{s'} \text{rawScore}_{s'}} \times 100
$$

The top server always receives 100. All others are expressed as a percentage of that.

## Imputation

Servers that did not participate in a test (failed preflight or did not run it) receive an **imputed** effective RPS for that test rather than zero. This prevents absent servers from being unfairly penalised for gaps in coverage.

The imputed value is the **30th percentile** of the participating servers' effective RPS for that *(test, VU level)*, anchored at zero:

$$
\text{effRPS}_{\text{imputed}} = P_{30}\bigl(\{0\} \cup \{\text{effRPS}_{s',t,v} : s' \in \text{participants}\}\bigr)
$$

Anchoring at zero means: the fewer servers participate in a test, the lower the imputed value. A server absent from a test it could have aced receives less benefit than one present with real results.

## Exclusions

A server is excluded from a test if it:

- returned HTTP `404` or `501` for the operation during preflight
- returned a semantically incorrect response during preflight (wrong resource type, missing required fields, incorrect code mapping)

Exclusions are recorded in the capability matrix and reflected in the imputation step rather than being treated as zero performance.
